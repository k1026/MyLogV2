# MyLog Agent Skills: Quick Reference

| Skill Name | 1-Line Summary (Role) | Usage Scene (When to use) |
| :--- | :--- | :--- |
| **@cording** | プロジェクトの標準規約と技術スタックを遵守させる | 開発の開始時や、基本構造・ディレクトリ設計を確認したい時 |
| **@compact** | 挨拶や解説を省き、修正後のコード差分のみを出力する | 構造を理解しており、とにかく速く実装コードだけが欲しい時 |
| **@focus** | 最小限のファイル読み込みで、特定の箇所を外科的に修正する | バグの特定や、巨大なファイルの一部だけをピンポイントで直す時 |
| **@strict** | 型定義の事前検証と、実装後のLint/型チェックを徹底する | 複雑なロジックの実装、DB定義の変更、手戻りを防ぎたい時 |
| **@tdd-first** | 仕様からテストコード（.test.ts）のみを先行生成する | 新機能の実装開始時、まず仕様をテストとして固めたい時 |
| **@tdd-fix** | テストログを解析し、パスさせるための実装を行う | `npm run test:ai` の結果を受け、最短でRedからGreenにしたい時 |
| **@tdd-strict** | 修正箇所の型安全性の最終確認と最適化を行う | テスト通過後、`any` の混入チェックやリファクタリングをしたい時 |

## TDD 標準ワークフロー
0. docs/specs/ 内の.md仕様書を読み込ませて要件を理解させる。このステップを独立で行うのが理想。
1. **@tdd-first** を実行し: AI にテストコードと関数の「空のシグネチャ」を作成させる。
2. **人間による実行**: ターミナルで `npm run test:ai` を実行し、結果をログに出力する。ログにエラーが無ければすり抜けている可能性があるので注意(エラーが無くてはならない)。実装と関係ない部分にエラーがでている場合も注意。
3. **@tdd-fix** を実行: AI が `test-results.log` を読み取り、エラーを元にAロジックを実装する。テストをパスするまで繰り返し。
4. **@tdd-strict** を実行: テストをパスしたあとAI に実装箇所の型安全性を検証させ、コードを綺麗に仕上げる。
